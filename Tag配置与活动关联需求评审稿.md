# Tag 配置与活动关联需求评审稿

**文档目的**：将当前 Tag 配置问题、现有数据结构、推荐解决方案及评审确认项整理成一份可直接用于需求评审的需求文档，帮助开发快速理解业务背景并讨论可行性方案。

**适用范围**：未来美术坊小程序管理端、活动打卡页、日常发帖页、用户动态打卡信息库、现有 Tag 库。

---

## 一、需求背景

当前业务中，Tag 主要用于两类场景：

1. **日常发帖 Tag**：用户在发布日常帖子时可勾选的标签，由管理端配置。
2. **活动打卡 Tag**：用户参与活动时，在过程打卡页必须选择的标签，来源于活动相关配置。

现阶段，管理员对 Tag 的管理存在多套来源：

- 日常发帖使用的 Tag，有单独的 **Tag 库** 维护，可配置：
  - Tag 名称
  - 权重（权重越高，前端展示越靠前）
  - 有效状态
- 日常发帖的内容数据，存放在 **用户动态打卡信息库**。
- 活动相关 Tag，则是在管理员新增活动信息时，由管理端 **自动生成**。

随着活动和内容逐渐增多，当前 Tag 已出现“来源分散、管理入口不统一、关联关系不清晰”的问题，需要在本期将其梳理清楚。

---

## 二、现状梳理

### 2.1 当前 Tag 类型

| Tag 类型 | 使用场景 | 当前来源 | 当前是否可统一配置权重/状态 |
|----------|----------|----------|-----------------------------|
| 日常发帖 Tag | 用户发布日常帖子 | 管理端 Tag 库手动配置 | 是 |
| 活动打卡 Tag | 用户参与活动过程打卡 | 创建活动时自动生成 | 否 / 不明确 |

### 2.2 当前涉及的数据载体

| 数据载体 | 当前作用 | 当前问题 |
|----------|----------|----------|
| Tag 库 | 存储日常发帖 Tag，支持权重、有效状态 | 目前未完整承接活动 Tag |
| 用户动态打卡信息库 | 存储用户发布的动态/打卡内容 | 已使用 Tag，但缺少统一 Tag 主数据来源 |
| 活动信息数据 | 存储活动基本信息 | 活动 Tag 来源于活动创建，但与 Tag 库关系不清晰 |

---

## 三、核心问题

### 3.1 业务问题

当前最核心的问题有三点：

1. **活动 Tag 是否应纳入管理端 Tag 列表统一管理？**
2. **活动 Tag 是完全由运营手动配置后再关联活动，还是活动创建后自动进入统一 Tag 列表？**
3. **如何在不推翻现有数据结构的前提下，让“用户动态打卡信息库”与“Tag 库”都继续发挥作用？**

### 3.2 现有痛点

- **管理入口不统一**：日常发帖 Tag 在 Tag 库中管理，活动 Tag 却从活动创建逻辑中产生，运营难以统一查看。
- **状态与排序能力不一致**：日常发帖 Tag 支持权重和有效状态，活动 Tag 是否能统一设置不明确。
- **关联关系模糊**：活动 Tag 是活动私有，还是平台公共可复用 Tag，当前没有标准。
- **后续维护成本高**：若继续分散管理，活动越多，Tag 越多，运营和开发都难以维护。

---

## 四、方案对比

### 4.1 方案 A：所有 Tag 都由运营在 Tag 列表手动配置，再手动关联活动

**方案说明**：
所有 Tag 都先由管理员在管理端 Tag 列表中手动创建，再在活动配置时选择要关联的 Tag。

**优点**：
- 管理入口统一
- Tag 主数据清晰
- 活动与日常发帖都可复用同一批 Tag

**缺点**：
- 运营配置步骤变多
- 活动创建流程变长
- 若活动很多，手动关联成本较高
- 与当前“活动创建即自动生成 Tag”的现状差异较大

### 4.2 方案 B：活动创建时自动生成活动 Tag，并自动同步到 Tag 聚合页统一管理（推荐）

**方案说明**：
保留现有“活动创建时自动生成活动 Tag”的机制，但自动生成后，系统将该 Tag 同步写入统一 Tag 库，并在管理端 Tag 聚合页中展示。运营可在统一页面继续配置其权重、有效状态，并查看该 Tag 的来源和所属活动。

**优点**：
- 保留现有活动创建逻辑，改动相对可控
- 管理端最终只有一个统一的 Tag 视图
- 日常发帖 Tag 和活动 Tag 都能配置权重、有效状态
- 便于开发复用现有 Tag 库与用户动态打卡信息库

**缺点**：
- 需要补充“活动 Tag 与活动 ID 的关联关系”
- 需要明确活动删除、活动修改时 Tag 的同步规则

### 4.3 方案 C：活动 Tag 继续独立存在，不进入 Tag 列表

**方案说明**：
维持现状，日常发帖 Tag 与活动 Tag 分开管理。

**优点**：
- 短期开发改动最小

**缺点**：
- 管理入口长期割裂
- 无法统一设置权重和有效状态
- 运营无法快速判断 Tag 来源与使用范围
- 后续扩展成本最高

### 4.4 方案结论

**推荐采用方案 B：活动创建自动生成 + 自动汇总到统一 Tag 聚合页。**

推荐原因：

- 对现有业务改动最小，便于平滑落地
- 能把现有 **Tag 库** 真正升级为“统一 Tag 主数据中心”
- 能让管理员在一个页面里统一管理权重、有效状态、来源和关联关系
- 既满足运营灵活配置，也便于开发沿用现有库表能力

## 五、推荐方案详细设计

## 5.1 设计原则

1. **统一管理入口，保留现有业务习惯**。
2. **Tag 库作为唯一 Tag 主数据来源**。
3. **活动 Tag 不再游离在活动配置之外，而是自动汇总进 Tag 聚合页**。
4. **用户动态打卡信息库继续存储业务内容数据，Tag 库负责存储 Tag 配置数据。**

## 5.2 数据角色划分

| 数据对象 | 推荐职责 |
|----------|----------|
| Tag 库 | 作为统一 Tag 主表，存储所有 Tag 的基础信息、类型、来源、权重、有效状态 |
| 用户动态打卡信息库 | 存储用户发帖/打卡内容，以及内容选择了哪些 Tag |
| 活动信息数据 | 存储活动基本信息，并记录活动与 Tag 的关联关系 |

### 5.3 推荐新增/补充字段

建议基于现有 Tag 库补充以下字段：

| 字段 | 说明 |
|------|------|
| tagId | Tag 唯一标识 |
| tagName | Tag 名称 |
| tagType | Tag 类型：`daily_post` / `activity_checkin` |
| sourceType | 来源类型：`manual` / `activity_auto` |
| relatedActivityId | 若为活动 Tag，则记录所属活动 ID；非活动 Tag 为空 |
| weight | 排序权重，数值越大越靠前 |
| status | 有效状态：启用 / 停用 |
| isDeleted | 删除标记 |
| createdAt / updatedAt | 创建时间 / 更新时间 |

> 若开发评估认为不适合直接改 Tag 库，也可采用“Tag 主表 + 活动 Tag 关系表”的形式实现，但管理端展示层仍需呈现为统一的 Tag 聚合页。

## 六、管理端需求

### 6.1 新增统一 Tag 聚合页

**管理端位置建议**：未来美术坊 → 基础配置 → Tag 列表

### 6.2 Tag 聚合页展示字段

| 字段 | 说明 |
|------|------|
| Tag 名称 | 标签名称 |
| Tag 类型 | 日常发帖 Tag / 活动打卡 Tag |
| 来源 | 手动创建 / 活动自动生成 |
| 所属活动 | 若为活动 Tag，则展示关联活动名称；否则为空 |
| 权重 | 用于前端排序 |
| 有效状态 | 启用 / 停用 |
| 创建时间 | Tag 创建时间 |
| 操作 | 编辑、启停、查看关联信息 |

### 6.3 管理规则

#### 6.3.1 日常发帖 Tag

- 由管理员在 Tag 列表中手动新增
- 支持编辑 Tag 名称
- 支持设置权重
- 支持设置有效状态
- 停用后，不再出现在前端发帖可选列表中
- 已历史使用的内容记录不回写删除，仅影响后续新内容发布

#### 6.3.2 活动打卡 Tag

- 管理员创建活动时，系统按活动配置自动生成活动 Tag
- 自动生成后，系统需同步写入 Tag 库
- 在 Tag 列表中自动展示，类型标识为“活动打卡 Tag”，来源标识为“活动自动生成”
- 运营可继续设置其权重和有效状态
- 不建议在 Tag 聚合页直接删除活动 Tag，避免与活动配置脱节
- 如需移除，建议通过活动侧取消关联或停用处理

### 6.4 筛选能力

Tag 聚合页需支持以下筛选条件：

- Tag 类型
- 来源类型
- 有效状态
- 所属活动
- Tag 名称搜索

这样可帮助运营快速区分“日常发帖 Tag”和“活动打卡 Tag”。

---

## 七、前端与业务规则

### 7.1 日常发帖页取数规则

前端发日常帖时，仅拉取：

- `tagType = daily_post`
- `status = 启用`

排序规则：

1. 按权重倒序
2. 权重相同按创建时间倒序或名称排序（由开发确认）

### 7.2 活动打卡页取数规则

用户进入某活动的过程打卡页时，仅拉取：

- `tagType = activity_checkin`
- `relatedActivityId = 当前活动 ID`
- `status = 启用`

排序规则同样按权重倒序。

### 7.3 用户动态打卡信息库使用方式

用户动态打卡信息库继续承担“内容事实数据”角色：

- 存储用户发帖/打卡内容
- 存储该条内容勾选的 Tag 信息（建议至少保留 `tagId`，如需兼容展示可冗余 `tagName`）

这样可以保证：

- Tag 的配置能力由 Tag 库统一维护
- 内容与 Tag 的使用记录仍保留在业务内容库中
- 历史内容不会因 Tag 被停用而丢失展示依据

---

## 八、关键同步规则

### 8.1 创建活动

当管理员创建活动并生成活动 Tag 时：

1. 生成活动 Tag
2. 写入 Tag 库
3. 记录 `tagType = activity_checkin`
4. 记录 `sourceType = activity_auto`
5. 记录 `relatedActivityId = 当前活动 ID`

### 8.2 编辑活动

若活动名称或活动 Tag 配置发生变化，需同步更新：

- 若 Tag 名称跟随活动名称，则自动同步 Tag 名称
- 若 Tag 允许独立编辑，则需明确以哪一方为准（建议评审确认）

### 8.3 删除活动

活动删除时，不建议物理删除历史 Tag 数据，建议规则如下：

- 关联活动 Tag 自动置为停用
- 在 Tag 聚合页可查看历史来源，但默认不在前端展示
- 若历史内容曾引用该 Tag，仍保留内容历史数据

### 8.4 停用 Tag

- 停用后，该 Tag 不再出现在前端可选列表中
- 已发布内容中已使用的 Tag，不做历史回删
- 活动打卡页若所有 Tag 都被停用，需要给管理端提示，避免前端出现“无 Tag 可选”的异常场景

---

## 九、为什么该方案能把现有两套库都用起来

### 9.1 Tag 库的作用

Tag 库继续保留，并从“日常发帖 Tag 配置库”升级为“统一 Tag 配置中心”，负责：

- 管所有 Tag 的名称
- 管权重
- 管有效状态
- 管类型
- 管来源
- 管活动关联关系

### 9.2 用户动态打卡信息库的作用

用户动态打卡信息库不需要被替代，继续负责：

- 存帖子/打卡内容本身
- 存用户实际选择了哪些 Tag
- 支撑内容详情、列表展示、统计分析

### 9.3 两者关系

可以理解为：

- **Tag 库 = 配置中心 / 主数据**
- **用户动态打卡信息库 = 业务内容事实数据**

两者分工明确，不冲突，也无需推翻现有结构。

## 十、需求评审时建议重点展示内容

建议在评审会上按以下顺序讲解：

### 10.1 先讲清现状

一句话说明：

> 当前 Tag 分为“日常发帖 Tag”和“活动打卡 Tag”两类，但来源不同、管理入口不同，导致运营无法统一配置和查看。

### 10.2 再讲清问题

一句话说明：

> 现在需要明确：活动 Tag 是否纳入统一 Tag 列表管理，以及怎样在不推翻现有库结构的情况下，实现统一配置权重、有效状态和关联关系。

### 10.3 最后讲推荐方案

一句话说明：

> 推荐保留活动创建自动生成 Tag 的现状，但把活动 Tag 自动汇总进统一 Tag 聚合页，让所有 Tag 最终都在一个管理入口中被配置和查看。

---

## 十一、评审需确认的问题

以下问题建议在需求评审会上明确拍板：

| 编号 | 评审问题 | 推荐默认方案 |
|------|----------|--------------|
| Q1 | 活动 Tag 是否统一纳入 Tag 列表管理？ | 是，统一纳入 |
| Q2 | 活动 Tag 采用手动关联还是自动生成后入库？ | 自动生成后入库 |
| Q3 | 活动 Tag 在 Tag 聚合页中是否允许直接删除？ | 不允许直接删除，建议停用或从活动侧解除 |
| Q4 | 活动名称变更时，Tag 名称是否自动同步？ | 建议自动同步，若 Tag 支持独立命名需额外说明 |
| Q5 | 历史内容已使用的 Tag，被停用后是否保留展示？ | 保留历史展示，仅影响新内容选择 |

---

## 十二、给开发讨论可行性时的落点

开发评审时建议重点讨论以下内容：

1. **现有 Tag 库是否可直接扩字段承接活动 Tag**
2. **是否需要单独补一张“活动-Tag 关系表”**
3. **活动创建、编辑、删除时，Tag 如何同步更新**
4. **历史内容中存 Tag ID、Tag 名称，还是两者都存**
5. **前端是否已具备按类型、状态、活动 ID 过滤 Tag 的接口能力**

---

## 十三、最终结论

本需求的核心不是重新造一套 Tag 体系，而是：

**在保留现有活动自动生成逻辑的基础上，把所有 Tag 汇总到统一 Tag 库中管理，让运营能统一配置权重和有效状态，让前端能按场景准确取数，让开发能在现有库结构上平滑扩展。**

该方案兼顾了：

- 运营可配置性
- 管理端统一性
- 现有数据结构复用
- 开发实施可行性

建议作为本次评审的默认推进方案。
